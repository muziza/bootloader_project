# Описание функций кода bootloader для STM32H7

## Назначение кода
Этот код реализует простой UART-bootloader для STM32H7. Он умеет:
- инициализировать GPIO и USART3;
- по кнопке оставаться в bootloader-режиме;
- стирать область приложения во Flash;
- записывать и читать прошивку по UART;
- проверять корректность приложения;
- передавать управление пользовательскому приложению.

Основная логика такая: после старта МК инициализирует GPIO и UART, затем проверяет кнопку и валидность приложения. Если кнопка не нажата и приложение корректно, управление сразу передаётся приложению. Иначе контроллер остаётся в bootloader-режиме и ждёт команды по UART.

## Глобальные константы и макросы

### Область приложения
- `APP_FLASH_BASE`, `APP_FLASH_END`, `APP_FLASH_SIZE` — адрес начала, конца и размер области Flash, где хранится пользовательское приложение.
- `APP_FIRST_SECTOR`, `APP_LAST_SECTOR` — номера секторов Flash, которые bootloader стирает под приложение.

### ОЗУ
- `SRAM_BASE_ADDR`, `SRAM_END_ADDR` — диапазон обычной SRAM.
- `AXI_SRAM_BASE_ADDR`, `AXI_SRAM_END_ADDR` — диапазон AXI SRAM.
- Используются для проверки, что начальный указатель стека приложения указывает в RAM.

### UART и протокол
- `UART_BAUDRATE` — скорость UART (`115200`).
- `FLASH_WORD_SIZE` — размер блока записи во Flash (`32` байта).
- `CMD_ERASE`, `CMD_WRITE`, `CMD_READ`, `CMD_JUMP` — коды команд bootloader-протокола.
- `ACK_OK`, `ACK_ERROR`, `ACK_UNKNOWN` — байты ответа на успешную, ошибочную или неизвестную команду.

### GPIO
- `GPIO_MODE_INPUT`, `GPIO_MODE_OUTPUT`, `GPIO_MODE_AF` — режимы ножек GPIO.
- `LED_PORT`, `LED_PIN` — светодиод.
- `BUTTON_PORT`, `BUTTON_PIN` — кнопка входа в bootloader.

### Flash
- `FLASH_KEY1`, `FLASH_KEY2` — ключи разблокировки Flash.
- `BL_FLASH_CR_*`, `BL_FLASH_SR_*` — локальные битовые маски для управления стиранием/записью Flash и проверки статуса.
- `BL_FLASH_ERROR_FLAGS`, `BL_FLASH_CLEAR_FLAGS` — наборы флагов ошибок и очистки состояния Flash.

### Совместимость USART
- Макросы `USART_ISR_RXNE_RXFNE` и `USART_ISR_TXE_TXFNF` нужны для совместимости между версиями CMSIS/STM32-заголовков.

## Функции

### `delay(volatile uint32_t count)`
Простейшая программная задержка на пустом цикле. На каждой итерации выполняется `__NOP()`.

### `gpio_set_mode(GPIO_TypeDef *port, uint32_t pin, uint32_t mode)`
Устанавливает режим работы выбранного GPIO-пина через регистр `MODER`. Используется как вспомогательная функция при инициализации ножек.

### `gpio_init(void)`
Инициализирует GPIO:
- включает тактирование портов `GPIOC`, `GPIOD`, `GPIOE`;
- настраивает LED как выход;
- настраивает кнопку как вход с подтяжкой;
- переводит `PD8` и `PD9` в альтернативную функцию USART3.

### `button_pressed(void)`
Читает состояние кнопки по `IDR`. Возвращает `1`, если кнопка нажата, и `0`, если не нажата.

### `led_toggle(void)`
Инвертирует состояние светодиода через `ODR`. Нужна для индикации активности bootloader.

### `uart3_init(void)`
Настраивает USART3:
- включает тактирование USART3;
- очищает `CR1`;
- вычисляет делитель `BRR` из `SystemCoreClock`;
- включает передатчик, приёмник и сам UART.

### `uart_rx(void)`
Блокирующий приём одного байта по UART. Ждёт, пока во входном FIFO/регистре появятся данные, затем возвращает принятый байт.

### `uart_try_rx(uint8_t *byte)`
Неблокирующий приём одного байта. Если данных нет, возвращает `0`; если байт принят — записывает его по указателю и возвращает `1`.

### `uart_tx(uint8_t byte)`
Блокирующая передача одного байта. Ждёт готовности передатчика и записывает байт в `TDR`.

### `uart_write(const uint8_t *data, uint32_t length)`
Передаёт массив байт по UART, по одному байту через `uart_tx()`.

### `uart_puts(const char *text)`
Передаёт C-строку до символа `\0`. Используется для текстового сообщения при старте bootloader.

### `uart_rx_u32(void)`
Принимает 4 байта по UART и собирает из них `uint32_t` в little-endian формате. Применяется для чтения `offset` и `length` в командах.

### `flash_wait(void)`
Ждёт завершения операции Flash, пока выставлены флаги занятости `BSY` или `QW`.

### `flash_unlock(void)`
Разблокирует Flash-банк 2 для операций записи/стирания, если установлен бит блокировки `LOCK`.

### `flash_lock(void)`
Снова блокирует Flash после завершения операций.

### `flash_clear_errors(void)`
Очищает флаги завершения и ошибок Flash через `CCR2`.

### `flash_erase_sector(uint32_t sector)`
Стирает один сектор Flash:
- проверяет номер сектора;
- ждёт готовности Flash;
- очищает старые ошибки;
- настраивает номер сектора и режим erase;
- запускает стирание;
- ждёт завершения и проверяет ошибки.

Возвращает `0` при успехе и `-1` при ошибке.

### `flash_erase_application(void)`
Стирает всю область приложения, проходя по секторам от `APP_FIRST_SECTOR` до `APP_LAST_SECTOR`. На время операции отключает прерывания, разблокирует Flash, стирает сектора, затем блокирует Flash обратно и инвалидирует instruction cache.

### `flash_write_block(uint32_t address, const uint8_t data[FLASH_WORD_SIZE])`
Записывает один блок размером 32 байта во Flash по указанному адресу.

Коротко по работе:
- проверяет, что адрес лежит в области приложения и выровнен на 32 байта;
- ждёт готовности Flash и включает режим программирования;
- записывает данные по 32 бита напрямую в память Flash;
- ждёт завершения операции;
- выключает режим программирования;
- перечитывает записанные байты и проверяет, что запись прошла правильно.

Возвращает `0` при успехе и `-1` при ошибке.

### `stack_in_ram(uint32_t stack)`
Проверяет, попадает ли значение указателя стека в один из допустимых диапазонов RAM: обычную SRAM или AXI SRAM.

### `app_is_valid(void)`
Проверяет, выглядит ли пользовательское приложение корректным.

Что именно проверяет:
- первый word вектора таблицы не равен `0xFFFFFFFF`;
- адрес начального стека лежит в RAM;
- адрес reset handler не равен `0xFFFFFFFF`;
- адрес точки входа лежит внутри области приложения;
- адрес точки входа имеет установленный младший бит, то есть это Thumb-адрес.

Если все проверки прошли, возвращает `1`, иначе `0`.

### `jump_to_application(void)`
Передаёт управление пользовательскому приложению.

Коротко по шагам:
- читает начальный `MSP` и адрес reset handler из векторной таблицы приложения;
- запрещает прерывания;
- выключает SysTick;
- отключает и очищает pending-состояния NVIC;
- переключает `VTOR` на таблицу векторов приложения;
- очищает/инвалидирует кэши;
- устанавливает новый `MSP`;
- выполняет `reset_handler()` приложения.

### `cmd_erase(void)`
Обработчик команды стирания. Вызывает `flash_erase_application()` и отправляет по UART `ACK_OK` или `ACK_ERROR`.

### `cmd_write(void)`
Обработчик команды записи прошивки.

Коротко по работе:
- принимает `offset` и `length`;
- проверяет, что запись не выходит за границы области приложения;
- отключает прерывания и разблокирует Flash;
- принимает данные по UART блоками по 32 байта;
- если последний блок неполный, дополняет его `0xFF`;
- записывает каждый блок через `flash_write_block()`;
- в конце блокирует Flash, включает прерывания и инвалидирует instruction cache;
- отправляет статус выполнения.

### `cmd_read(void)`
Обработчик команды чтения.
- принимает `offset` и `length`;
- проверяет границы области приложения;
- при ошибке сразу отвечает `ACK_ERROR`;
- при успехе отправляет `ACK_OK`, затем выдаёт запрошенные байты из Flash по UART.

### `cmd_jump(void)`
Обработчик команды запуска приложения.
- сначала проверяет приложение через `app_is_valid()`;
- при ошибке отвечает `ACK_ERROR`;
- при успехе отправляет `ACK_OK`, дожидается окончания передачи по UART и вызывает `jump_to_application()`.

### `process_command(uint8_t cmd)`
Центральный диспетчер команд bootloader. По значению `cmd` вызывает один из обработчиков: erase, write, read, jump. Если команда неизвестна, отправляет `ACK_UNKNOWN`.

### `bootloader_mode(void)`
Основной цикл bootloader-режима.

Что делает:
- отправляет строку `BOOTLOADER READY`;
- бесконечно мигает светодиодом;
- периодически проверяет, не пришёл ли байт команды по UART;
- если команда пришла, передаёт её в `process_command()`.

### `main(void)`
Точка входа программы.

Порядок работы:
1. Инициализирует GPIO.
2. Инициализирует USART3.
3. Если кнопка не нажата и приложение валидно, сразу переходит в приложение.
4. Иначе остаётся в `bootloader_mode()`.

## Как работает код целиком

### Режим старта
После включения bootloader всегда поднимает периферию, нужную для выбора режима: кнопку, светодиод и UART. Дальше он решает, запускать приложение сразу или перейти в режим ожидания команд.

### Переход в приложение
Если кнопка bootloader не нажата и во Flash есть корректный образ приложения, bootloader не делает лишней работы и сразу прыгает в пользовательский код. Перед прыжком он аккуратно переносит таблицу векторов, стек и системное состояние.

### Работа в bootloader-режиме
Если приложение отсутствует, битое или пользователь удерживает кнопку, контроллер остаётся в bootloader. В этом режиме хост по UART может отправлять команды на стирание, запись, чтение и запуск прошивки.

### Протокол обмена
Протокол очень простой: хост посылает один байт команды, а дальше — параметры и данные, если команда их требует. В ответ bootloader отправляет один байт статуса, а для чтения ещё и сами данные.

### Защита от ошибок
В коде есть базовые проверки границ памяти, адресов и результата записи во Flash. Это защищает от записи вне области приложения, от прыжка в невалидный код и от частично повреждённого образа.
